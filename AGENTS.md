# 协作约定

本仓库默认使用中文沟通。除非文件上下文或外部接口明确要求英文，否则说明、注释、变更说明与协作过程中的文字输出应优先使用中文。

代码修改应始终以逻辑正确为前提。在此基础上，再追求结构清晰、命名准确、边界稳定、实现可维护。任何整理、重构、迁移或模块调整，都不应为了“更好看”而改变既有行为；如果行为必须变化，应明确说明原因、影响范围与验证方式。

## 代码与结构

本仓库当前采用多模块结构，核心边界如下：

- `cli/`：命令行入口，仅负责用户交互、命令组织与调用 `:core`
- `core/`：业务能力层，负责对上层暴露稳定能力边界
- `dexkit/`：KMP 包装层，负责提供 `io.github.dexclub.dexkit` 这一层 API
- `dexkit/vendor/DexKit`：vendored 上游 DexKit
- `dexkit/vendor/libcxx-prefab`：Android 构建链依赖的本地 `libcxx` prefab 仓库

修改前应先理解当前职责边界，避免跨层直接调用、重复封装或绕过现有抽象。涉及 `cli / core / dexkit` 的调整时，应优先保持依赖方向清晰，避免把平台实现、构建细节或第三方细节泄漏到不该出现的层级。

整理代码时，应优先提升可读性与可维护性，避免无必要的抽象、兼容层、工具类堆叠和过度拆分。对非显而易见的实现，可以添加少量高价值注释；不要添加重复代码表面含义的注释。

## 依赖与构建

新增依赖、替换依赖或升级依赖时，应遵循“必要且最小”的原则。若当前仓库已有实现路径或已有依赖足以完成目标，不应额外引入新库。

任何较大改动完成后，都应进行最小可用验证。至少验证本次改动直接影响到的模块、任务或构建链，而不是只完成代码修改而不做构建确认。当前仓库常用验证方式包括但不限于：

```bash
./gradlew :dexkit:compileKotlinJvm
./gradlew :dexkit:assembleAndroidMain
./gradlew :core:compileKotlinJvm :cli:compileKotlin
./gradlew :cli:fatJar
```

如果改动影响 Android 构建链，应同时确认本地 `mavenLocal()`、`NDK`、`cmake`、`ninja` 以及 `libcxx` 发布状态是否与当前环境要求一致。

构建相关改动应尽量避免引入只对当前机器成立的硬编码。如果因为当前维护环境必须这样处理，应在注释或 README 中明确标注其适用范围，避免将环境特例误写成通用方案。

新增脚本、任务或构建开关时，应优先保证名称直观、职责单一，并避免留下只服务一次性排障的临时逻辑。临时方案若被保留，必须能说明其存在理由。

## 文件与编码

本仓库直接维护的文件统一要求使用 `UTF-8` 且无 `BOM`。若新增或修改文件，应保证编码符合该要求。

对于 vendored 上游仓库中的文件，应尽量保持原样：不主动调整编码、不顺手格式化、不做与当前目标无关的风格性修改。只有在完成当前任务确有必要时，才对上游文件做最小修改。

不应提交构建产物、缓存文件、临时文件、本地环境垃圾或与任务无关的生成目录。涉及 submodule 时，应清楚区分“主仓库改动”和“子仓库改动”，避免把两者混成一次不清晰的提交。

## 文档与交付

如果仓库结构、模块关系、构建方式、环境前提或开发流程发生变化，应同步更新 README 或相关说明，保证文档与当前代码状态一致。

README 应优先承担“项目概览、上手入口、常用命令”的职责，不应逐步膨胀成开发日志、实现流水账或设计文档镜像。README 中也不应专门设置“设计文档”之类的小节去承载或汇总实现细节导航；更细的规则、边界、执行流与实现细节应放入 `.docs/` 或专门说明文档，而不是持续堆入 README。

删除旧代码、旧目录、旧配置或旧构建逻辑时，应先确认仓库内已经没有实际引用，尤其注意 Gradle 任务、资源路径、脚本入口和文档引用，避免只删除表面文件而留下隐性依赖。

提交说明应优先参考仓库现有提交历史的风格，尤其保持与最近几次提交在语言、粒度和表述方式上的一致性。没有特殊理由时，不应突然切换成完全不同的命名或描述风格。

在提交或交付结果时，应优先给出实际变更、验证结果与剩余风险；如果存在仅适用于当前维护环境的前提，也应明确标注，避免将环境特例误写成通用结论。
