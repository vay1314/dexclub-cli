# DexClub MCP V3 运行时可观测性

## 目标

本稿只定义 `mcp` 运行期的可观测性边界，回答以下问题：

1. 终端默认应输出哪些最小日志
2. 哪些高频轨迹应进入详细日志，而不是刷到终端
3. JVM / native 崩溃证据应如何保留
4. 运行时日志开关的职责边界应如何划分

本稿不重复讨论：

- MCP tool 面与字段契约
- skill 编排规则
- release / packaging 结构

这些内容见：

- `43-mcp-tool-surface.md`
- `44-skill-workflow.md`

## 设计原则

`mcp` 是一个长期运行的工具服务，不是交互式 CLI。

因此运行期日志策略不应走两个极端：

- 终端完全静默，导致出错时黑盒
- 所有请求轨迹都刷到终端，导致正常使用时噪音过大

更合理的分层是：

- 终端：最小运行日志
- 文件：详细轨迹日志
- JVM / native：崩溃文件

一句话说：

- 终端负责“当前是否正常”
- 详细日志负责“出了问题如何回看”
- 崩溃文件负责“进程为什么直接死”

## 终端最小日志

终端默认应始终保留以下最小日志：

### 1. 启动成功

至少输出：

- 监听地址
- 当前是否开启详细轨迹
- crash 文件默认写入位置

例如：

- `DexClub MCP listening on http://127.0.0.1:8787/mcp ...`

### 2. tool failure

当某个 tool 执行失败时，终端应直接可见：

- tool 名
- 错误类型
- 错误摘要

这里不应要求用户先去翻文件。

`tool failure` 同时也应进入详细日志文件，以便保留完整排障上下文。

### 3. 未捕获异常

若运行时出现未捕获异常，终端应直接输出：

- thread
- 当前 operation
- exception class
- message

### 4. 进程退出

当进程关闭时，终端应至少输出：

- shutdown 事件
- 当前 operation（若有）

这样可以区分：

- 人工 `Ctrl+C`
- 正常停止
- 某次处理中退出

## 详细日志文件

以下高频轨迹不应默认刷到终端，而应进入详细日志文件：

- HTTP request
- HTTP response
- HTTP failure
- MCP tool begin
- MCP tool end
- 参数摘要
- 请求耗时

这些日志的价值是：

- 事后排障
- 对齐 agent 行为
- 判断问题出在握手、tool 调用还是底层执行

但它们的缺点也很明显：

- 数量多
- 正常运行时噪音大
- 容易淹没真正异常

因此更适合写入文件，而不是长期占据终端。

## 详细日志文件位置与命名

详细日志文件不应使用时间目录。

当前阶段更合理的固定结构是：

```text
logs/mcp.log
logs/mcp.log.1
logs/mcp.log.2
logs/mcp.log.3
logs/mcp.log.4
logs/mcp.log.5
```

含义如下：

- `logs/mcp.log`
  - 当前活跃写入文件
- `logs/mcp.log.1 ~ .5`
  - 按新到旧保留的历史轮转文件

这样做的原因是：

- 当前文件路径固定，用户容易找到
- 不需要先猜日期或实例目录
- 对本地单机工具场景足够清晰

## 详细日志文件轮转策略

详细日志文件应按**大小**轮转，而不是按时间轮转。

推荐默认值：

- 单文件目标上限：`10 MB`
- 历史文件上限：`5`

总量上限大致为：

- `mcp.log + 5` 个历史文件
- 约 `60 MB`

之所以不优先采用按时间轮转，是因为 `mcp` 的日志流量是突发型的：

- 平时可能很安静
- 出问题时几秒内可能产生大量轨迹

按时间切分容易出现：

- 安静时切出很多很小的文件
- 高峰时单文件仍然可能过大

按大小轮转更符合当前排障需求。

## 软上限与事件边界

详细日志文件的大小阈值不应作为硬截断边界。

更合理的语义是：

- `10 MB` 是目标上限，不是逐字节硬切
- 一条完整日志事件应先写完
- 写完后若文件已超过阈值，则在**下一条事件写入前**轮转

这里的“完整日志事件”包括：

- 单条普通日志
- 一整段异常堆栈

这样可以保证：

- 不会把同一条堆栈拆进两个文件
- 日志边界可读
- 排障时不需要跨文件拼接半条异常

一句话说：

- 大小阈值是软上限
- 轮转发生在事件之间，而不是事件中间

## 日志文件内容范围

详细日志文件应优先覆盖以下字段：

### HTTP 层

- method
- uri
- json-rpc id
- rpcMethod
- tool（若为 `tools/call`）
- contentType
- accept
- session 相关标识
- elapsedMs

### tool 层

- tool begin
- tool end
- tool failure
- 当前关键参数摘要

参数摘要应保持克制：

- 只记录高价值键
- 避免无上限记录全文本参数
- 避免把大文本导出结果本身写入日志

## crash 文件策略

普通日志无法覆盖 JVM / native 直接崩溃。

因此 `mcp` 运行期还应保留：

- `hs_err_pid*.log`
- `*.hprof`（例如 OOM）

这些文件的职责是：

- 当进程没有机会打印 `failure` 日志时，仍保留致命崩溃证据

它们不属于终端日志，也不应混入常规 MCP 详细轨迹文件。

## 默认行为

推荐的默认行为是：

### 终端

- 最小运行日志：默认开启

### 文件

- 详细日志文件：默认开启，可通过显式开关关闭
- 开启后写入固定位置：
  - `logs/mcp.log`
- 并按大小轮转：
  - `10 MB`
  - `最多 5 个历史文件`

### crash 文件

- 默认开启

这里的关键不是“默认越多越好”，而是：

- 正常使用时不过度打扰
- 出问题时又不至于完全黑盒

## 开关语义

运行期日志开关不应只叫 `DEBUG`，因为它承载的并不只是调试模式，而是：

- 详细轨迹记录
- tool 调用可见性
- 参数摘要输出

因此更合理的语义是：

- 终端最小日志：常开
- 详细日志文件：默认开启，可显式关闭

如果沿用环境变量，则其职责应表达为：

- 是否关闭高频详细轨迹文件
- 而不是是否“允许错误被看见”

错误、未捕获异常与 shutdown 不应依赖详细日志开关。

## 不建议的方向

当前不建议：

### 1. 默认把所有 HTTP / tool 轨迹都打到终端

这样会：

- 噪音过高
- 掩盖真正异常
- 降低日常使用可读性

### 2. 完全关闭终端异常信息，只写文件

这样会：

- 让用户误以为服务无响应
- 出错时仍然像黑盒

### 3. 为日志策略引入过重的抽象

例如一开始就做：

- 多文件轮转
- 多级 logger 体系
- 复杂 sink 配置

当前阶段更重要的是：

- 先把最小层级和职责边界定清楚

## 一句话结论

DexClub MCP 的运行时可观测性应采用：

- `终端最小日志 + 详细日志文件 + JVM/native crash 文件`

而不是：

- 完全静默
- 或终端全量刷屏
