# DexClub V3 Skill 工作流设计

## 目标

本稿只回答以下 5 件事：

1. skill 在 DexClub AI 能力体系中承担什么角色
2. skill 应如何围绕 MCP tool 组织多步分析
3. skill 在什么情况下应定位、检查、取证、回退
4. skill 应如何区分线索、事实、证据与结论
5. skill 应遵守哪些编排约束

本稿不重复讨论：

- `core / mcp / skill` 的总体边界
- MCP P0 tool 的字段与输入输出契约

这些内容见：

- `42-ai-capability-surface.md`
- `43-mcp-tool-surface.md`

## role 定位

skill 不是：

- 底层能力提供者
- CLI 包装器
- MCP 协议适配器

skill 是：

- 分析编排层

它的职责是：

1. 理解用户问题
2. 抽取当前问题中的线索
3. 决定下一步调用哪个 tool
4. 判断当前结果属于线索、事实还是证据
5. 在必要时回退到另一条路径继续分析
6. 汇总结论并说明依据

一句话说：

- skill 负责“怎么分析”
- mcp 负责“能调用什么工具”
- core 负责“底层能力真实存在且稳定”

## skill 的基本工作方式

skill 不应把分析任务理解为：

- 一次调用拿答案

更合理的理解是：

- 一轮多步迭代，逐步收敛

这条工作流不应被实现成：

- 固定顺序的硬编码状态机

更合理的理解是：

- 一组可回退、可跳步、可重入的编排规则

典型过程如下：

1. 从用户问题中抽取已知线索
2. 选择一条最低成本但足以启动的定位路径
3. 得到候选对象后，缩小范围
4. 对关键对象做一层检查
5. 必要时导出原始证据
6. 若当前路径证据不足，则回退并尝试另一条定位路径
7. 当证据足以支持判断时，再形成结论

## skill 的输入理解

用户输入给 skill 的，不一定是：

- 完整方法名
- 完整类名
- 已知入口

更常见的是：

- 功能描述
- 一段文案
- 一个资源名
- 一条行为现象
- 一段竞品证据
- 一句“为什么会这样”

因此 skill 第一件事通常不是直接检查方法，而是：

- 抽取可用线索

常见线索类型包括：

- 字符串
- 资源名
- manifest 组件
- 类名片段
- 方法名片段
- 已知对象引用

## 四类输出的使用规则

skill 必须区分四类对象：

1. 线索
2. 事实
3. 证据
4. 结论

### 线索

线索用于：

- 启动分析
- 缩小范围

线索并不等于：

- 已确认实现点

### 事实

事实用于：

- 描述对象的一层稳定属性

例如：

- 某方法有哪些 callers
- 某方法用了哪些 strings
- 某方法有哪些 annotations

事实不等于：

- 已经解释了为什么会这样

### 证据

证据用于：

- 支撑或否定某个分析假设

例如：

- 某方法的 smali
- 某类的 java 视图

### 结论

结论只能由 skill 基于：

- 线索
- 事实
- 证据

整合后产生。

skill 不应把：

- 单个字符串命中

直接升级成：

- 功能实现位置已确认

## skill 的通用动作流

以下流程表示：

- 常见推荐路径

而不是：

- 所有任务都必须严格按同一顺序执行

skill 应根据当前问题、已有对象和证据强度，决定：

- 是否跳过某一步
- 是否重复某一步
- 是否直接回退到更早阶段

### 1. 定位

当当前问题仍然缺少稳定对象时，skill 应先走定位动作。

常见优先使用：

- `find_methods_using_strings`
- `find_classes_using_strings`
- `manifest`
- `find_res`
- `resolve_res`
- `list_res`

skill 在这一阶段的目标不是拿结论，而是：

- 找到值得继续看的候选簇

### 2. 缩小

当定位结果返回多个候选对象时，skill 应继续缩小，而不是立刻导出大文本。

缩小策略包括：

- 优先类级候选，再缩到方法
- 优先和当前问题线索最强相关的对象
- 优先返回字段更完整的候选

### 3. 检查

当 skill 已经有较可信的目标方法时，应先走一层检查，而不是立刻拉原始证据。

常见优先使用：

- `inspect_method`

检查阶段的目标是回答：

- 这个对象是否值得继续下钻
- 当前问题需要的关键一层事实是否已经存在

### 4. 取证

当一层事实还不足以支撑结论，或者需要消歧时，skill 才应进入取证阶段。

常见优先使用：

- `export_method_java`
- `export_class_java`
- `export_method_smali`
- `export_class_smali`

这里的关键不是固定顺序，而是：

- 当前问题需要哪种视图

一般可理解为：

- `java`
  - 更适合快速建立高层理解
- `smali`
  - 更适合做真实性确认和消歧

### 5. 回退

当当前路径出现以下情况时，skill 应回退：

- 候选命中太弱
- 一层事实不足以支持当前假设
- 原始证据否定了当前推断
- 当前路径开始消耗大量成本但没有明显增益

回退不是失败，而是正式工作流的一部分。

## skill 的视图选择原则

skill 不应机械遵循：

- 永远先看 `java`
- 永远先看 `smali`

更合理的原则是：

1. 先选择最适合当前问题的视图
2. 当当前视图不足以支撑判断时，再切换另一种视图
3. 不为了少一次返回而强迫走不合适的视图

因此：

- `java` 与 `smali` 都是正式视图
- 选择由问题驱动，而不是由工具偏好驱动

## skill 的会话使用方式

skill 应优先使用：

- `session`
- `method_handle`
- `class_handle`

而不是每一步都重传：

- `workdir`
- 完整 descriptor

原因不只是：

- 省 token

还包括：

- 减少状态漂移
- 减少对象识别不一致
- 让多轮分析保持连续性

### 会话中的推荐做法

1. 一轮分析开始时先拿到 `session`
2. 候选阶段尽量保留 handle
3. detail / export 阶段优先基于 handle 展开
4. 若切换到另一 target，再显式开启或切换另一 session

更准确地说，skill 的默认工作方式应是：

- `session-first`

也就是：

- 能基于 session 工作时，不再回退到无状态 `workdir` 路径

`workdir` 直调更适合作为：

- 启动阶段
- fallback
- 尚未建立 session 的过渡路径

## skill 的终止条件

skill 不应无限展开。

一轮分析应在以下任一条件满足时收束：

1. 已找到足以支撑问题回答的证据链
2. 当前问题所需的最小事实已经明确
3. 所有合理路径都无法继续提供更强证据
4. 当前剩余不确定性必须由人工上下文补充

### 收束时应明确区分

- 已确认结论
- 高概率推断
- 仍待验证点

## skill 的输出约束

skill 面向最终用户输出时，应优先提供：

1. 结论
2. 关键依据
3. 剩余不确定性

如果引用到 MCP 返回结果，应明确说明它属于：

- 线索
- 事实
- 原始证据

避免把所有命中结果混写成一段模糊描述。

## skill 的 P0 约束

P0 skill 工作流不应做以下事情：

- 直接依赖 CLI shell
- 假设所有 MCP tool 都可无约束并发
- 自动展开 deep trace / graph
- 一拿到候选就立刻导出整批大文本
- 把单个命中直接升级成结论

P0 skill 应优先做到：

- 能稳定完成一轮定位 -> 检查 -> 取证 -> 结论的闭环

## 一句话结论

skill 的核心价值，不在于“多会调几个 tool”，而在于：

`把线索、事实、证据组织成一轮可回退、可收束、可解释的分析过程`
